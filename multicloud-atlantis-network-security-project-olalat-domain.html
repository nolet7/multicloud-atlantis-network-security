Enterprise Multi-Cloud Atlantis Project for olalat.xyz

This project builds an enterprise-grade Terraform and Atlantis platform for a large multinational company operating across AWS, Azure, and GCP. It provisions cloud networking and security controls through reusable modules, uses a remote state backend at the correct stage, and enforces no-local-apply operations through pull requests and Atlantis.

1. Program introduction and enterprise architecture

Business goal

Provide a centralized Infrastructure as Code control plane for multinational branches so that networking and security baselines are consistent in every region and cloud.

What Atlantis does

Atlantis runs terraform plan and terraform apply from pull requests. Engineers do not apply from laptops. Plans are reviewed, approved, and executed centrally.

Cloud scope

AWS creates VPCs and security groups, Azure creates VNets and NSGs, and GCP creates VPCs and firewall rules. A common environment model controls naming and sizing.

High-level flow

Engineer updates Terraform code
        ↓
Pull request opened in GitHub
        ↓
Atlantis receives webhook at https://atlantis.olalat.xyz/events
        ↓
Atlantis runs terraform plan on changed projects
        ↓
OPA / Conftest / tflint / tfsec checks policies
        ↓
Reviewers approve
        ↓
Authorized approver comments: atlantis apply
        ↓
Terraform applies using remote state backend
        ↓
Network and security controls deployed consistently

cat > modules/aws-network/main.tf <<'EOF'
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

resource "aws_vpc" "this" {
  cidr_block           = var.vpc_cidr
  enable_dns_support   = true
  enable_dns_hostnames = true
  tags = merge(var.tags, {
    Name        = "${var.name_prefix}-vpc"
    Environment = var.environment
  })
}

resource "aws_security_group" "web" {
  name        = "${var.name_prefix}-web-sg"
  description = "Enterprise web security group"
  vpc_id      = aws_vpc.this.id

  ingress {
    description = "HTTPS from approved ranges"
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = var.allowed_ingress_cidrs
  }

  egress {
    description = "Restricted outbound"
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = var.allowed_egress_cidrs
  }

  tags = merge(var.tags, {
    Name        = "${var.name_prefix}-web-sg"
    Environment = var.environment
  })
}

output "vpc_id" { value = aws_vpc.this.id }
output "security_group_id" { value = aws_security_group.web.id }
EOF

cat > modules/azure-network/main.tf <<'EOF'
terraform {
  required_providers {
    azurerm = {
      source  = "hashicorp/azurerm"
      version = "~> 4.0"
    }
  }
}

resource "azurerm_resource_group" "this" {
  name     = "${var.name_prefix}-rg"
  location = var.location
  tags     = merge(var.tags, { Environment = var.environment })
}

resource "azurerm_virtual_network" "this" {
  name                = "${var.name_prefix}-vnet"
  address_space       = [var.vnet_cidr]
  location            = azurerm_resource_group.this.location
  resource_group_name = azurerm_resource_group.this.name
  tags                = merge(var.tags, { Environment = var.environment })
}

resource "azurerm_network_security_group" "web" {
  name                = "${var.name_prefix}-nsg"
  location            = azurerm_resource_group.this.location
  resource_group_name = azurerm_resource_group.this.name
  tags                = merge(var.tags, { Environment = var.environment })
}

resource "azurerm_network_security_rule" "https_in" {
  name                        = "AllowHttpsInbound"
  priority                    = 100
  direction                   = "Inbound"
  access                      = "Allow"
  protocol                    = "Tcp"
  source_port_range           = "*"
  destination_port_range      = "443"
  source_address_prefixes     = var.allowed_ingress_cidrs
  destination_address_prefix  = "*"
  resource_group_name         = azurerm_resource_group.this.name
  network_security_group_name = azurerm_network_security_group.web.name
}

output "resource_group_name" { value = azurerm_resource_group.this.name }
output "vnet_name" { value = azurerm_virtual_network.this.name }
output "nsg_name" { value = azurerm_network_security_group.web.name }
EOF

cat > policies/deny_public_ssh.rego <<'EOF'
package terraform.security

deny[msg] {
  rc := input.resource_changes[_]
  rc.type == "aws_security_group"
  ingress := rc.change.after.ingress[_]
  ingress.from_port == 22
  ingress.to_port == 22
  ingress.cidr_blocks[_] == "0.0.0.0/0"
  msg := "Public SSH is not allowed in AWS security groups"
}
EOF

cat > policies/require_tags.rego <<'EOF'
package terraform.governance

required_tags := {"Owner", "CostCenter", "DataClass", "Branch"}

deny[msg] {

  rc.change.after.tags
  missing := required_tags - object.keys(rc.change.after.tags)
  count(missing) > 0
  msg := sprintf("Missing required tags: %v", [missing])
}
EOF

4. Remote tfstate setup at the right stage

Why tfstate must come early

Remote state is the control plane record of your infrastructure. In enterprise environments, state must be secured before live deployments start. This prevents drift, accidental overwrites, and competing applies.

CloudBackendWhy
AWSS3 + DynamoDBS3 stores encrypted state and DynamoDB handles locking to prevent two applies at once.
AzureAzure Storage containerBlob container provides centralized state with Azure RBAC and storage protections.
GCPGCS bucketGCS provides durable remote state with IAM controls and versioning.

Bootstrap commands

# AWS tfstate
aws s3 mb s3://olalat-terraform-state --region us-east-1
aws s3api put-bucket-versioning --bucket olalat-terraform-state --versioning-configuration Status=Enabled
aws dynamodb create-table   --table-name olalat-terraform-locks   --attribute-definitions AttributeName=LockID,AttributeType=S   --key-schema AttributeName=LockID,KeyType=HASH   --billing-mode PAY_PER_REQUEST   --region us-east-1

# Azure tfstate
az group create --name olalat-tfstate-rg --location eastus
az storage account create --name olalattfstate001 --resource-group olalat-tfstate-rg --location eastus --sku Standard_LRS
az storage container create --name tfstate --account-name olalattfstate001

# GCP tfstate
gcloud storage buckets create gs://olalat-terraform-state --location=us-central1
gcloud storage buckets update gs://olalat-terraform-state --versioning

5. Reusable Terraform modules

Modules are the enterprise standardization layer. They hide repeated cloud details, force naming consistency, and make branch rollouts predictable.

  • modules/aws-network: VPC and security group baseline
  • modules/azure-network: Resource group, VNet, NSG baseline
  • modules/gcp-network: VPC and firewall baseline

Why modules matter in multinational companies

  • Every country branch gets the same security starting point.
  • Policy enforcement becomes easier because resources follow the same shape.
  • Rollouts can expand from one region to many without redesigning everything.

7. Deploy a GCP VM to host Atlantis

Use a dedicated GCP VM for Atlantis so the service is centralized, reachable from GitHub webhooks, and easier to secure behind DNS and TLS.

Create the VM

gcloud compute instances create atlantis-vm   --project YOUR_GCP_PROJECT_ID   --zone us-central1-a   --machine-type e2-standard-2   --subnet default   --tags atlantis,http-server,https-server   --image-family ubuntu-2204-lts   --image-project ubuntu-os-cloud   --boot-disk-size 30GB

Open firewall for web access

gcloud compute firewall-rules create allow-atlantis-https   --network default   --allow tcp:80,tcp:443,tcp:4141   --target-tags atlantis

Install Docker and Atlantis

ssh YOUR_USER@ATLANTIS_VM_PUBLIC_IP

sudo apt update
sudo apt install -y docker.io docker-compose nginx certbot python3-certbot-nginx git
sudo usermod -aG docker $USER
newgrp docker

git clone https://github.com/nolet7/multicloud-atlantis-network-security.git
cd multicloud-atlantis-network-security/atlantis
docker compose up -d

8. DNS, TLS, and Atlantis URL access using olalat.xyz

DNS record to create in Porkbun

Create an A record for atlantis.olalat.xyz pointing to the public IP of your GCP Atlantis VM.

TypeHostValueTTL
AatlantisYOUR_GCP_VM_PUBLIC_IP300

NGINX reverse proxy

sudo tee /etc/nginx/sites-available/atlantis.olalat.xyz > /dev/null <<'EOF'
server {
    listen 80;
    server_name atlantis.olalat.xyz;

    location / {
        proxy_pass http://127.0.0.1:4141;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
EOF

sudo ln -s /etc/nginx/sites-available/atlantis.olalat.xyz /etc/nginx/sites-enabled/atlantis.olalat.xyz
sudo nginx -t

12. Validation and troubleshooting

Quick validation checklist

  • Can you open https://atlantis.olalat.xyz?
  • Does GitHub webhook delivery show HTTP 200?
  • Does Atlantis comment on pull requests?
  • Can each environment plan independently?
  • Is remote state created in the right backend per cloud?

Common issues

IssueLikely causeFix
Webhook failsWrong secret or DNS/TLS issueConfirm webhook secret and test /events path.
Atlantis URL not openingFirewall or NGINX proxy issueCheck GCP firewall, nginx config, and Docker port 4141.
State lock stuckInterrupted runInspect backend lock and use unlock procedures carefully.
Prod apply deniedMissing approval or branch behind mainGet approval and rebase the PR.

Final enterprise recommendation

Start with one business unit and one region, validate the module and policy model, then replicate the pattern to other branches. For a multinational rollout, keep modules shared, keep live roots environment-specific, and keep Atlantis centralized behind atlantis.olalat.xyz.
